# Time for Family (TFF)

Single-file Python + wxPython "cozy creature-park sim." Players adopt and care for animals across multiple rooms (indoor / outdoor / aquatic / aviary / modder-defined types), watch them mature, pair, breed, age into elders, and produce items. The plugin suite of associated synthesis tools (rozaya-jsfx, the eventual Audio Park successor) lives in separate repos.

## Layout

The game is now **seven files** (was one ~11.8k-line `.pyw`). The brain runs headless; the screen is split into readable pieces. Import direction is strictly one-way and acyclic: `time_for_family.pyw` → `tff_panels` / `tff_editors` → `tff_dialogs` → `tff_sound` → `tff_engine` → `tff_announcements`. Each UI module pulls the names it needs from the modules below it via `import X` + `globals().update({k:v for k,v in vars(X).items() if not k.startswith("__")})` — this copies even the single-underscore helpers a plain star-import skips, and the **shared shelves come across as references to the SAME objects**, so an in-place reload anywhere is visible everywhere. **Never rebind a shelf with `=`; mutate it** (`.clear()`/`.update()`/`[:]=`).

- `tff_engine.py` — the **game brain** (~3,750 lines): world state, creatures, breeding, aging, rooms, time, save/load, content loading (`load_types`/`load_text_assets`/`ensure_user_data_dir`), the text/name helpers (`random_creature_name`, etc.), and the **player-action cores** (`rename_creature`, `add_villager`, `set_auto_breeding`, `apply_settings`, `plan_room_retype`/`apply_room_retype` — see the "Player action cores" section). **No wxPython, no sound, no NVDA** — imports and runs fully headless. This is what a headless AI player drives.
- `tff_announcements.py` — the **words** (~480 lines): `DEFAULT_ANNOUNCEMENTS` (every player-visible line), `_ANNOUNCEMENT_DOCS`, the live `ANNOUNCEMENTS` shelf, `format_announcement`, and the announcements-file load/write helpers. **Zero imports, zero engine dependency** — `load_announcements(text_dir)` takes the text directory as an argument so the file is standalone (a wording bug is contained here). `ANNOUNCEMENTS` is the **same dict object** across engine + every UI module (verified).
- `tff_sound.py` — **sound + NVDA** (~140 lines): chime synthesis (`synthesize`/`ensure_sounds`/`play_sound`) and the NVDA controller client (`nvda_speak`). Imports only `PROJECT_DIR`/`SOUNDS_DIR` from the engine. The mute flag (`SOUND_MUTED`) lives here with its only reader `play_sound`; the UI toggles it through `set_muted()`/`is_muted()` (a direct `global SOUND_MUTED` in the UI would update a stale copy).
- `tff_dialogs.py` — the **everyday pop-ups** (~1,325 lines): the gameplay dialogs (`AddVillagerDialog`, `EditRoomDialog`, `ExpandSlotDialog`, `BuildRoomDialog`, `HelpDialog`, `RenameDialog`/`prompt_rename`, `SettingsDialog`) plus the **shared checklist helpers** (`make_state_announcing_checklist` & friends) used by both the everyday dialogs and the editors. Depends only on engine + sound — **no panel, editor, or MainFrame reference** (keeps the graph acyclic).
- `tff_editors.py` — the **modding editors** (~2,460 lines): the heavy content-authoring dialogs (`SpeciesEditorDialog` + `SpeciesDialog` curator, `RoomTypeEditorDialog` + `ManageRoomTypesDialog`, `ManageAnnouncementsDialog`) plus their sub-widgets (`DisabilityListEditor`, `MeterPanel`, `IngredientPanel`) and the pure state helpers they use to count/purge content on delete (`_slugify_id`, `_species_in_use_count`, `_room_type_in_use_count`, `_purge_species_from_state`, `_purge_room_type_from_state`). Depends on engine, sound, and `tff_dialogs` (for the shared checklist helpers). MainFrame opens the three top-level ones; the editors never reference panels.
- `tff_panels.py` — the **four notebook pages** (~1,680 lines): `RoomPanel`, `VillagePanel`, `ParkPanel`, `StatsPanel`. Each opens dialogs from `tff_dialogs`; none reference each other or MainFrame (MainFrame builds them).
- `time_for_family.pyw` — the **window + entry point** (~1,260 lines): `MainFrame` and `main()`. Launch this file to play. Holds the MainFrame-only runtime toggles `PAUSED` / `AUTO_BREEDING` / `AMBIENT_ENABLED`.
- `tff_play.py` — the **"play by typing" layer** (Phase 3): a **text adventure** over `tff_engine`. The front door is `command(save_path, text)` — it parses one plain-English line ("adopt cat", "move Mittens to Cozy Room", "care for Cozy Room", "look", "help") into a verb + args and dispatches to the verb functions (`look`/`adopt`/`dig`/`build`/`move`/`care`/`breed`, plus the later-added `rename`/`expand`/`convert`-room-type/`autobreed`/`reset`/`reload` and care-one-creature), each of which loads the save, advances time, acts via the engine cores, saves, and returns a plain-English string. (`reload` re-reads the cached species/room/text definitions from disk — for when content is hand-edited mid-session; saves are re-read every command regardless.) Forgiving parsing (articles/prepositions stripped, numbers picked out, unknown commands answered with a hint — never an error). `save_path` is first so each player/AI has its own park. No wx. The hearthkin side is **one** tool, `tools/creature_park.py` in the hearthkin checkout (`creature_park(command)`), enabled per-kin in `~/.hearthkin/kin/<kin>/tools.json`; **no `exec` capability needed**. See the Phase-3 session note.
- `dev/headless_smoke.py` — no-windows regression test for the **engine**. `python dev/headless_smoke.py`: loads content, adopts, saves/reloads, ages, breeds, builds a room + decays its meters, exercises the player-action cores, and asserts the "names don't leak between species" canary + that importing the engine pulls in **no** wx. `dev/ui_smoke.py` — builds the real **windows** headlessly: constructs `MainFrame` + all panels against a temp copy of the save, refreshes, then builds **every dialog** (including the modding editors and their sub-widgets — the stuff easy to never open in play) so construction-time and file-split breakage is caught without clicking. Run **both** smokes after any structural change. `dev/split_engine.py` / `dev/split_announcements.py` / `dev/split_screen.py` / `dev/split_editors.py` are the one-off splitters that produced the files (kept for reference; do not re-run on the already-split tree).
- `assets/sounds/` — chimes and ambient SFX. Always read from here.
- `assets/text/`, `assets/types/` — **factory** copies of all modder-editable content. Read-only after first launch. Players who want to revert any species/room-type/text-pool to its shipped default copy the file back from `assets/` into `user_data/`.
- `user_data/text/`, `user_data/types/` — **live** copies. On first launch `ensure_user_data_dir()` copies the assets/ trees here; from then on every read and write goes through user_data/. The in-game editors (File → Species, Mods → Manage room types, the rename-and-save flow, etc.) ONLY touch user_data/. Path constants `TEXT_DIR`, `TYPES_DIR`, `SPECIES_DIR`, `ROOM_TYPES_DIR` all point under user_data/. `ASSETS_TEXT_DIR` / `ASSETS_TYPES_DIR` are the read-only originals; only `ensure_user_data_dir()` reads them.
- `state.json` — saved game state (auto-saved as the player plays). Always at the project root.
- `docs/MODDING.md` — modder-facing reference. Keep in sync when species/room-type schemas change.

**5 species ship loaded by default:** cat, dog, rabbit, chicken, fish. (The other 8 mentioned in older session notes — hamster, bird, mouse, snail, hedgehog, brownie, sprite, mossling — have leftover text-pool directories under `user_data/text/species/` but **no JSON specs on disk** and no UI to install them. The species-extras bundle helpers + the `Mods → Add or remove extra species` dialog were removed in May 2026 session 4 as YAGNI — the user found them confusing and the bundles never got authored. If those species are wanted later, drop a JSON spec into `user_data/types/species/` and they'll appear in the Species dialog like any other.) 5 shipped room types: indoor, outdoor, aquatic, aviary, glade.

**Empty-by-default first launch + single Species dialog.** A brand-new save (no rooms, no village, no baskets, no remembered creatures) auto-opens the **SpeciesDialog**: list of loaded species, detail panel, four actions — Bring [species] home, Edit, Delete, Create a new species. The dialog stays open after each action so the player can keep curating without re-opening it. Re-openable any time from **File → Species**, and fires automatically again after **File → New game**. This single dialog replaced three older ones in May 2026 session 4 (the original WelcomeDialog, ManageSpeciesDialog, and ExtraSpeciesDialog) — the user reported all three doing the same thing was confusing. Anti-completionism design: no "install all," no counts, single-species detail-on-select. Pre-seeding the village with babies of every species was removed because it pulled toward completionism (see `feedback_tff_no_completionism.md` in user-memory) and crowded shared room types with multiple species (see `feedback_tff_species_per_room.md`).

Once this becomes a git repo, ordinary commits replace that workflow (commit the whole tree as the baseline).

## Critical user context — read this before designing any UI

These come directly from the user's stated needs and from sessions where ignoring them caused real harm:

- **NVDA is the user's primary navigation method.** Every UI must work via keyboard + screen reader. Sliders should be `wx.SpinCtrl` / `wx.SpinCtrlDouble` / `wx.TextCtrl` for numeric input, not `wx.Slider` (Slider doesn't announce value changes on Windows under NVDA reliably, and rapid arrow-key holds flood the announce queue). Buttons use `&Letter` mnemonics.
- **Cogacc (cognitive accessibility) is non-negotiable.** Plain language, ≤5 things per visible screen, every input field labeled, no jargon, **no pipe-formats in user-facing UI**. Symmetric add/remove. Help text in low-contrast gray below or above input groups (see `compat_intro` and the Disabilities pool intro for the existing pattern).
- **Disability is a respectful representation feature, not a debuff.** This is in the codebase comment at `maybe_disability()` and is design-load-bearing. A creature with a disability can pair, breed, raise babies, take affection, age, and be adopted exactly like any other creature. **No mechanical penalty applies**, ever. The only place `cat["disability"]` is consulted (now or in future code) is auto-emigration: disabled creatures stay in the village rather than auto-leaving for the wild. Modders write descriptions in `disabilities.txt`; shipped species default to `disability_chance: 0` and an empty pool. **Don't link disability to inbreeding-as-debuff framing** — the inbreeding multiplier raises the *roll rate* but doesn't change what disability *means*.
- **Don't AppendText per streaming chunk** — same NVDA flooding cascade issue as Hearthkin. TFF doesn't currently stream LLM output, but if it ever does, buffer and paint atomically. (See hearthkin's CLAUDE.md for full context.)
- **No more notebook tabs.** The room navigator is now a `wx.ComboBox` (read-only, type-ahead) above a `wx.Simplebook`. Page changes flow through `MainFrame._select_book_page` which uses `book.ChangeSelection` (not `SetSelection`) so it doesn't yank focus to the new page. The combo-handler also re-focuses the picker via `wx.CallAfter` after each page swap so arrow-keying through rooms doesn't kick the user out. **Don't reintroduce a Notebook here** — Tab-strip navigation through 12+ rooms was the cog-acc problem this redesign solved.
- **All time is real time, no game-days.** The game-day abstraction was removed mid-2026 because it was confusing for no real benefit. Every age field on creatures and species is in real seconds (`age_seconds`, `elder_age_seconds`, etc.). Old saves and pre-redesign species JSONs auto-migrate via the backward-compat readers (`cat_age_seconds`, `species_elder_age_seconds`, `species_old_age_seconds`, `species_starter_age_range_seconds`). The legacy `seconds_per_game_day` setting is kept for migration only — don't expose it in Settings.
- **Aggregated NVDA announcements.** Any system that fires events about multiple creatures at once (life-stage transitions, wild emigration, sanctuary arrivals, settled, basket-ready) MUST aggregate into a single line when more than one fires in the same check. The pattern: pass a list to a `_announce_*` helper that picks `_one` vs `_many` template variants. Per-creature announcements at scale flood NVDA's UIA queue and freeze the screen-reader, just like the AppendText-per-chunk problem.
- **Don't push the player toward content completionism.** No "install all" buttons, no completion counters ("X of Y species installed"), no progress bars on optional content. The SpeciesDialog and any future content-picker UI should feel like invitations, not a checklist. The user has an OCD-flavored "must take them all home" pull that turns optional rosters into obligations and overwhelms them. Show one option at a time, make the picker easy to dismiss without feeling incomplete, and don't surface counts of un-installed content.
- **Mixed-species rooms are bad UX for this user.** Don't design features that encourage one room to hold multiple species. Prefer narrow `compatible_room_types` per species; if revisiting the model, lean toward species-specific room subtypes ("rabbit hutch") rather than broad shared rooms. The previous pre-seeded village + every-species-in-outdoor pattern was actively distressing.

## Design values to preserve

- **Cozy by default — no debuffs.** TFF's design philosophy is generous: creatures live full lives, every animal you adopt thrives, and the game doesn't punish neglect catastrophically. Avoid mechanics that introduce "things go wrong if you don't engage perfectly."
- **Modder-friendly storage.** All species, room types, and text pools are plain JSON / line-per-entry text files. Modders are first-class users — every UI editor writes back to those files, and players can hand-edit if they prefer.
- **Default species use cats as the canonical legacy.** Function names like `random_cat_name`, default test paths around cat species, and many comments treat cat as "the original." When extending, support all species symmetrically rather than special-casing cat.

### June 2026 session 1 — Phase 3: an AI can play (tff_play.py + hearthkin park tools)

Built the "play by typing" layer so an LLM (Tensor, via hearthkin) can play. Two design calls the user made: Tensor plays its **own separate park** (own save in its kin dir, never the human's), and the interface is **a few clear verbs + a look** (chosen over one fuzzy "do" tool or a dozen granular ones).

**TFF side — `tff_play.py`:** plain-English out, validated structured commands in, over the wx-free engine. Verbs `look(save_path, focus="")`, `adopt(species)`, `dig(times)`, `build(room_type, name)`, `move(creature, destination)`, `care(room)`, `breed(room)`. Each takes a `save_path` first (per-player save), `_load`s it (running `apply_elapsed_time` so the world stays alive between turns), acts through the engine's pure cores, saves, and returns prose. Forgiving name resolution (species/room/creature by name, case-insensitive), and every failure explains itself in words (wrong species → lists the loaded ones; can't afford a build → `format_shortfall`; breed not ready → maps the 8 `attempt_breed` statuses to friendly lines). Narration helpers describe creatures ("Pasillyn - a content baby female cat (newborn)") and rooms ("Cozy Room (an indoor room) - 1 of 4 slots filled … Care: Food full, Water full."). Regression guard: `dev/play_smoke.py` drives the whole loop headlessly against a temp save (13 checks, all pass).

**hearthkin side — `tools/creature_park.py` in the hearthkin checkout (ONE tool):** `creature_park(command: str)`, registered in `tools/__init__.py`'s `_REGISTRY`. A hearthkin tool is just a typed function with a docstring (schema auto-derived); `agent_name` is framework-injected + hidden from the model (same pattern as `note.py`), which is how the tool knows whose park to load. It locates the TFF install with **no machine-specific path baked in** (both pieces ship publicly — a hardcoded path would leak the username AND break for everyone else; see `feedback_no_machine_paths_in_shipped_code` in user-memory). `_game_dir()` resolves in order: `HEARTHKIN_TFF_PATH` env var → `~/.hearthkin/creature_park_path.txt` (one line, per-machine, never shipped) → home-dir conventions (`~/tff`, `~/time-for-family`, …) → a copy bundled beside the tool; clear error if none has `tff_play.py`. On the user's box `~/tff` matches with zero config. Then it puts that dir on `sys.path`, `import tff_play` lazily (only when actually called — the registry import never pulls the game in), and calls `tff_play.command(<kin dir>/creature_park.json, command)`. The save uses `Path.home()`; the engine uses `Path(__file__).parent` — all portable.

**Design correction (the user's call):** this started as seven separate `park_*` tools ("a few clear verbs"), but the user pointed out that's strange for a model that already does text-in/text-out — they expected a text adventure. Right call: collapsed to ONE `creature_park` command tool. The seven verbs still exist as the validated cores inside `tff_play`; the model just sees one tool and types commands at it. (The seven `park_*.py` files + the `_creature_park.py` helper were deleted.) Lesson: for an LLM-driven game, a single command tool reads more naturally — and is likely *easier* for a small model — than a menu of structured tools.

**The `exec` question (the user's):** No — playing the game does **not** require enabling hearthkin's `exec` tool (the approval-gated arbitrary-shell-command capability). `creature_park` is an ordinary tool, enabled per-kin in `tools.json`. Only the lazy "shell out to launch the game" approach would need exec, and we didn't take it. Verified Tensor's `tools.json` has no `exec`. Enabled `creature_park` there for Tensor (additive; remove the name to turn off). Tensor must restart hearthkin to load the new tool module (the enabled-list is re-read per chat, but tool code loads at startup).

**Verified:** the bridge runs end-to-end (a hearthkin tool helper → `tff_play` → engine → plain English back, with a per-kin save created in the kin dir); all hearthkin tool files compile; `tff_play` doesn't break the game (additive module, the `.pyw` doesn't import it); headless + play + ui smokes all pass; windowed launch clean.

**Verb set expanded (same session, user asked for "everything the UI can do" on the gameplay side):** added `rename` (rename_creature), `expand` (slot expansion, charges `slot_expansion_common_cost` common items), `convert` (room-type change via plan/apply_room_retype, relocating creatures that no longer fit), `autobreed on/off` (set_auto_breeding), `reset` (new_state, two-step: bare `reset` warns, `reset confirm` wipes), and `care for <creature>` (pets one via pet_cat). All ride engine cores that already existed from the de-tangle. `dev/play_smoke.py` covers them.

**Content authoring deliberately NOT exposed.** The species / room-type / announcements editors have no pure cores AND they write to the **shared** content under `user_data/` (species/room-type/text JSONs, announcements) — which the human's game also reads. So a kin authoring content would change the human's game too. The user's instinct (correct): give each kin its **own full data sandbox** (its own `user_data` copy, seeded from the shipped `assets/`), not just its own save, so content authoring is safe and isolated — then expose the editors. That needs the engine's data root (`TEXT_DIR`/`TYPES_DIR`/`SPECIES_DIR`/`ROOM_TYPES_DIR` + STATE_FILE) to become per-kin configurable (currently all anchored to `PROJECT_DIR`, the code dir; only STATE_FILE is overridden today, by `tff_play._load`). `ASSETS_*`/`SOUNDS_DIR` stay at PROJECT_DIR (factory/code, shared, read-only). Not yet built — the next step if the user wants kin-authored content.

### June 2026 session 1 — audit of the (now-isolated) dialogs + editors

The user distrusted the dialogs because they'd never played through many of them. With the editors isolated in `tff_editors.py`, ran two parallel read-audits (`tff_editors.py` + `tff_dialogs.py`) focused on the Save/Validate/Delete handlers — the logic a construct-test can't reach. The everyday dialogs (`tff_dialogs.py`) came back essentially clean (all the session 4–6 fixes hold: float-clamp exemptions, duration `max(1,…)`, blank-vs-zero, treasure re-validation, shared-shelf mutation, accessible TextCtrl inputs). The editors turned up two real bugs, both fixed and verified:

- **MAJOR — editing a room type silently dropped `treasure_cost`.** `RoomTypeEditorDialog.on_save` rebuilt the spec from scratch with only six keys, so any field the editor doesn't expose vanished — and the Glade ships `treasure_cost: 1`. Editing+saving the Glade made it free to build; any future room-type field would die the same way. Fix: seed the save dict from the originally-loaded spec (`self._loaded_spec`, stashed in `_build`), overwrite only the editor-owned keys, then `pop("compatible_species")` to keep dropping just the legacy field (species `compatible_room_types` stays the single source of truth). Verified headlessly: glade's `treasure_cost` now survives into the save dict.
- **MEDIUM — deleting a species wiped a text folder shared with another species.** `text_directory` is a deliberately-shareable field; `SpeciesDialog.on_delete` `rmdir`'d it unconditionally, stripping a co-sharing species' name/description/colour pools. Fix: compute `text_dir_shared` (any other loaded species resolving to the same slugged folder); skip the folder deletion when shared (only the JSON goes), and tell the player so in the confirmation.

**Deferred (lower-priority, real but edge-case or ambiguous — pick up next):**

### May 2026 session 6 — offline-birth age frozen at 0

User reported a village dog showing "Age 9 minutes; In village for 37 minutes" — visible contradiction. One-line change at the call site, all the logic lives in `baby_to_cat`.

Only way to land in that state is the bug we just fixed, so the snap is safe. - Player builds a room mid-gestation that solves the no-room problem → next tick's `check_expecting_room_space` sees room available, flips `no_room_warned` back to False; warning won't refire unless space disappears again.
- Two pending pairs of the same species, one room with one slot free → `check_expecting_room_space` simulates accumulated occupancy; one gets the all-clear, the other gets warned. Right outcome.

### May 2026 session 5 — gestation phase (chunk 1 of basket rework)

User wanted basket realism: advance warning before babies arrive, per-species lifecycle differences. This chunk lands the gestation/expecting phase. Per-species `basket_dependency_seconds` (chunk 2), no-room warning during gestation (chunk 3), and `independent_at_birth` (chunk 4) are deferred — each is independently shippable.

**The new flow:**
1. Pair breeds → if `gestation_seconds` > 0, an *expecting* record goes into `state["expecting"]`. No basket yet. Babies are pre-rolled at conception (so color inheritance, disability rolls, etc. are locked to the parents who actually conceived) and stored on the record.
2. `process_expecting()` runs each tick (live) and during offline catch-up. When a record's `due_at` passes, it converts to a basket. The basket's `created_at` is set to the original `due_at` so the basket-maturity timer counts from the moment of birth, not the moment the engine got around to processing it (matters for offline — a baby due 2 hours ago should be openable, not starting fresh).
3. Existing basket flow takes over from there: open, decide, babies become creatures, grow up.

**Backward compatible.** Default `gestation_seconds` is 0 — shipped species (cat / dog / rabbit / chicken / fish) and any pre-feature mod specs keep the old immediate-basket behavior with zero migration. The `gestation_seconds` field is in `SpeciesEditorDialog`'s Advanced section; modders opt in by setting a duration.

**`attempt_breed`** now returns `("conceived", expecting_record)` for gestating species or `("success", basket)` for instant-basket species. Also takes an optional `now=` parameter so offline catch-up can pass simulated time and gestation timing lines up with when the conception "happened" inside the away window.

**`auto_breed_offline_catchup`** walks the away window in `interval`-sized steps with a simulated clock. Each step: `attempt_breed(now=sim_now)` for every room, then `process_expecting(now=sim_now)` to mature anything whose due_at fell inside this iteration. At the end, conceptions still pending (gestation extends past the catchup window) get stashed on `state["_offline_conceptions"]` for the relaunch announcer; conceptions that both conceived and matured during the window are filtered out so they only get the basket-arrival announcement, not the conception one.

**`apply_elapsed_time`** also calls `process_expecting` independently in case `AUTO_BREEDING` is off (the catchup function early-returns and doesn't process_expecting). This covers the player who manually bred a pair, closed the game, and came back after gestation completed — they still get the basket-arrived announcement.

**New announcement events** (with per-event placeholder docs in `_ANNOUNCEMENT_DOCS`):
- `breed_conceived` — manual breed, gestating species
- `auto_breed_conceived_one` / `_many` — auto-breed live tick
- `auto_breed_conceived_offline_one` / `_many` — auto-breed during away period

**New announcers on MainFrame:**
- `_announce_offline_births` — surfaces baskets matured by the standalone `process_expecting` call (the AUTO_BREEDING-off path; matures pre-existing pending records)
- `_announce_offline_conceptions` — surfaces still-pending expecting records that started during the away period

**Cleanup:** `_purge_species_from_state` now also strips expecting records of the deleted species. Without this, when gestation completed the basket would be born with babies whose species no longer existed, and downstream code (basket dialog, `baby_to_cat`, etc.) would break or produce phantom-species creatures.

**Edge cases handled:**
- Expecting record's room is deleted before birth → basket arrives with stale `room_id`; `show_basket` already handles missing rooms (falls back to first room or shows friendly message). Nothing to redirect at purge time.
- Parent dies (emigrates) during gestation → babies are still in the expecting record with `parent_ids` stamped; basket arrives normally. Cozy framing: the babies are still due.
- `state_is_fresh` and `new_state` and `load_state` all updated to handle the new `expecting` key. Pre-feature saves get an empty list via `setdefault`.

**Suggested next steps for the rework** (each independently shippable):
- Chunk 2 — per-species `basket_dependency_seconds` to replace the global `mature_seconds` for openability wait
- Chunk 3 — periodic "no room has space for {species}" warning during gestation so the player can build/expand before the basket arrives
- Chunk 4 — `independent_at_birth: true` for precocial species (chickens, fish, snails); babies skip the basket entirely and arrive directly in a compatible room

### May 2026 session 5 — NVDA basket-dialog cutoff (text wrap clipping)

User reported NVDA stopping at "unchecked babies" mid-sentence in the BasketDialog instructions, with the rest of the text only reachable via word-by-word navigation. First guess (NVDA being interrupted by the 500ms timer's status updates) was wrong — the actual cause was visual clipping.

The instructions field was a `wx.TextCtrl` with `TE_READONLY | TE_MULTILINE | TE_NO_VSCROLL` and a fixed `size=(-1, 48)`. The text wrapped to ~3 visual lines but only 2 fit in 48px. NVDA's line-by-line navigation respected the visible bounds and stopped at the wrap-break, even though the underlying widget held the full text (word-by-word can dig deeper into the buffer past the visible region).

**Fix:** switched the instructions block from `wx.TextCtrl` to `wx.StaticText` with `.Wrap(640)` and the existing low-contrast gray foreground color. Same pattern already used by the room-edit "Allowed species" intro and other instruction blurbs in the codebase. StaticText doesn't clip, doesn't scroll, and reads naturally as part of NVDA's dialog scan.

**Also tried but didn't fix the issue** (kept the changes anyway because they're improvements):
- `_update_status` now caches and compares before each `ChangeValue` / `Enable`, so no-op ticks are truly silent (was firing accessibility events on every 500ms tick even when nothing changed).
- Countdown coarsened to whole minutes when more than 60 seconds remain (`"about 5 minutes"` rather than `"5m 23s"` updating every second). Last-minute precision drops back to seconds for the player to see the final countdown.
- Timer slowed from 500ms → 1000ms — half the rate of accessibility-event churn for no perceptible benefit (the displayed countdown can change at most once per second).

**Known risk pattern, not eradicated:** the codebase has ~30 instances of `wx.TextCtrl` with `TE_READONLY | TE_MULTILINE | TE_NO_VSCROLL` and small fixed heights, used as static help/intro/instruction blocks. Most are probably fine because their text is short enough to fit, but any one of them could clip if the text grows or the wrap point lands wrong. **For new instruction blocks, use `wx.StaticText` with `Wrap()` instead.** When fixing a reported clipping issue, convert the specific TextCtrl rather than tweaking heights — StaticText is the right tool for unchanging prose. (Reserve TextCtrl for genuinely-dynamic text where NVDA needs to know about value changes — heading_ctrl, status_text, preview_text in BasketDialog all stay TextCtrl.)

### May 2026 session 5 — pairs survive room transfers

The user clarified they did want pairs to persist across moves (earlier "I don't even care" was a tech-language overload, not a real opt-out). Switched the model from "every move dissolves the pair" to "pairs are forever, until death."

- **`move_creature_to_room` and `move_creature_to_village`** no longer touch `pair_id`, `pair_progress`, or `last_breed_per_pair` on the moved creature OR the partner. Reuniting them in the same room re-enables breeding without a fresh 30-minute bonding period. Cooldowns also persist with the pair (a couple that just bred and got moved together to a roomier room is still resting from breeding, not magically refreshed).
- **`_purge_room_type_from_state`** also stopped clearing `pair_id` on displaced creatures. If a partner was in a room of a different type (not being purged), the pair survives across the type-delete: one half goes to village, the other stays in their room, both still paired.

- **Sanctuary-move (disabled retiree → village) no longer dissolves pairs.** Same reasoning as a normal move-to-village. The retired disabled creature keeps their pair_id; the partner anywhere keeps theirs. Only true wild emigration (death) calls `_release_partner`.
- **Village-emigration now also calls `_release_partner`.** Previously it didn't, but it didn't need to either — village creatures used to all have `pair_id=None`. With pairs surviving moves, a village creature can hold a `pair_id` pointing at a partner in a room. When the village creature emigrates, that partner needs to be freed.

Design ethos: pairs are "till death do us part" — matches the cozy framing better than "moved them = different couple now," and removes a 30-minute re-bonding penalty for any room shuffle. There's no UI to manually break a pair; if one comes up later, that's a small future addition.

## Conventions for adding things

- **Reading or writing player-editable files:** use the path constants `TEXT_DIR`, `TYPES_DIR`, `SPECIES_DIR`, `ROOM_TYPES_DIR`. They all point under `user_data/`. Never construct paths under `assets/` for writes — `assets/` is read-only after first launch. The only legitimate readers of `ASSETS_TEXT_DIR` / `ASSETS_TYPES_DIR` are `ensure_user_data_dir()` (for the first-launch copy) and any future "reset to factory" feature.
- **New tunable setting:** add to `DEFAULT_SETTINGS` dict near top with a comment explaining what it does. If runtime-adjustable, also add an entry to `SettingsDialog.INT_FIELDS` or `FLOAT_FIELDS` with a plain-language label (under ~80 chars). Avoid SpinCtrl for the dialog input (NVDA flood); the existing `_add_int` / `_add_float` use plain TextCtrl already.
- **New species property:** add to the species spec JSON (default in the `else` branch of `SpeciesEditorDialog.__init__` where the new-species default spec lives), expose in `SpeciesEditorDialog` with a label + StaticText help below the input group, save in the editor's save handler. If it's a duration, use `add_duration` (not `add_int`) so modders can type "5 minutes" instead of seconds. Decide whether the field belongs in the **basic view** (top of the dialog, parented to `scroll`) or the **Advanced collapsible** (parented to `adv_window`). Default to Advanced — the basic view is deliberately small. The `add_text` / `add_int` / `add_duration` / `make_group` helpers all accept an optional `parent=` kwarg for this; pass `parent=adv_window` for advanced fields.
- **New disability flag:** the parser at `parse_disability_entry` recognizes `ok` and `no_produce`. To add another: extend the recognized set, add a checkbox to `DisabilityListEditor._add_row`, update `GetValue` to emit the new flag, update the help text in the species editor and the disabilities.txt header in `_TEXT_FILE_HEADERS`. Don't add flags that gate existence-of-creature behavior — flags should be small behavioral nuances, not "creature can't do X."
- **New text pool for a species:** add filename + header to `_TEXT_FILE_HEADERS`, add to the species editor's pool list (`add_pool` calls), include in the save loop's `text_files` list, load in `load_text_assets()` for the species. The pattern is consistent across the existing pools (names, descriptions, pet responses, disabilities).
- **New announcement event:** add to `DEFAULT_ANNOUNCEMENTS` dict AND `_ANNOUNCEMENT_DOCS` dict (placeholders documentation). When the event is about multiple creatures, ship `_one` and `_many` variants, plus `_offline_one` / `_offline_many` if it can fire from offline catch-up. Wire through a `_announce_*` helper on MainFrame that picks the variant. Existing modders' `announcements.txt` files keep working — new keys auto-resolve to the new defaults via `format_announcement`'s fallback.
- **New age field on creatures or species:** ALWAYS use `_seconds` suffix. ALWAYS read via the helpers (`cat_age_seconds`, `species_*_seconds`, `species_old_age_seconds`, `_spec_seconds`) rather than `cat["age_seconds"]` directly — the helpers handle backward-compat with old `_days` saves and the elder/retire merge. Add a one-shot `*_announced` flag on the creature if you need to fire an announcement on transition; backfill the flag in `load_state._migrate_age` so old saves don't flood.
- **Adding default species:** drop the JSON spec in `assets/types/species/<id>.json` AND `user_data/types/species/<id>.json` (the latter for existing players who already have user_data populated). Drop the text-pool directory in BOTH `assets/text/species/<id>/` AND `user_data/text/species/<id>/` with the standard files (`names_male.txt`, `names_female.txt`, `descriptions.txt`, `pet_responses.txt`, `disabilities.txt`, `colors.txt`). The species automatically appears in the WelcomeDialog picker on next launch. **Don't add it to a village seed list** — there is no village seed any more (`new_state` is empty). The player picks it themselves.
- **Cogacc check before shipping any UI change:** would NVDA announce this clearly? Are all controls labeled? Is there a pipe-format or jargon term that needs replacing? Are there more than 5 distinct elements competing for attention? Could focus get stuck somewhere unexpected (especially after a Simplebook page change or a dialog close)? If any answer is no, redesign before merging.
